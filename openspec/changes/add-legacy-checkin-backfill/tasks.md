## 3. Job queue: enqueue on login + worker loop (api)

- [x] 3.1 Add a `legacy_backfill_jobs` collection/repository: `{ _id, org_id, app_user_id, status: pending|active|done|failed, attempts, next_attempt_at, locked_at, last_error, created_at, updated_at }`; unique index on `app_user_id` — `db/legacy_backfill_jobs.rs` + indexes in `db/mod.rs::ensure_indexes`
- [x] 3.2 In `app_auth::login`, after the session is created and the response is assembled: if `org.legacy_backfill()` is `Some` and `user.legacy_backfill_done_at` is `None`, upsert a `pending` job for this `app_user_id` (no-op if one already exists) — single fast Mongo write, no `tokio::spawn`
- [x] 3.3 Extract `startup::repair_one` into a reusable, independently-callable function (`pub`, doc-commented) so the worker can call it after a job's backfill completes
- [x] 3.4 Implement the worker loop: started once at boot (`tokio::spawn` in `main.rs`), polling every 10s. Each tick: `recover_stale` then drains all due jobs via atomic conditional claim (`pending`+`next_attempt_at<=now` → `active`); runs provider + `repair_one`; on success sets `done` + `AppUser.legacy_backfill_done_at`; on failure increments `attempts`, computes `next_attempt_at` via capped backoff (60s/300s/1800s/3600s), or `failed` once `attempts >= 5` — `services/legacy_backfill/worker.rs`
- [x] 3.5 Confirmed login latency/behavior is unaffected by the enqueue write, and that job processing never happens inline within a request handler — verified via manual smoke test (6.3): login returned immediately, backfill completed asynchronously seconds later via the worker
- [x] 3.6 Handle worker-restart recovery: a job left `active` past a staleness threshold (5 min) is treated as abandoned and reset to `pending` on the next tick — `recover_stale` in `worker.rs`, backed by `LegacyBackfillJobRepository::recover_stale`
