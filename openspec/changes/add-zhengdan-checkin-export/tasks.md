Real-machine testing surfaced and fixed several environment-specific bugs along the way (TLS 1.2, `Invoke-RestMethod`/`Content-Type` charset mojibake, `.ps1` script-source-encoding mojibake, invalid `New-ScheduledTaskSettingsSet` params, `RepetitionDuration` out-of-range) — all documented in README.md and this file's earlier entries.
  - **Finding from initial real-machine testing (2026-07-12)**: downloading `export.ps1` via `Invoke-WebRequest` on the actual customer Windows Server 2016 box failed with `無法建立 SSL/TLS 的安全通道` — that .NET Framework defaults to SSL3/TLS1.0, not TLS 1.2, which both GitHub and the production API require. Fixed by adding `[Net.ServicePointManager]::SecurityProtocol = [Net.SecurityProtocolType]::Tls12` as the first executable line of `export.ps1` (so the scheduled runs are unaffected regardless of how the script was originally downloaded), and documented the one-off manual workaround needed for the download step itself in the README's new "PowerShell 5.1 的 TLS 1.2 陷阱" section. On-site smoke itself is still not done — this only fixes a blocker discovered while getting the file onto the machine.
  - **Finding (2026-07-12, continued)**: after seeding real check-in data and re-running against a live (tunneled) instance, display names came back as mojibake (e.g. "陳聖夫" → "é³èå¤«"), reproducible even when explicitly re-opening the output file as UTF-8 in Notepad. Root cause: `Invoke-RestMethod` on PowerShell 5.1 decodes the HTTP response body using a guessed non-UTF-8 encoding whenever the server's `Content-Type` doesn't declare a charset — our API returned bare `application/json`. Fixed on both sides: (1) `api/src/handlers/checkin_export.rs`'s `export` handler now returns `Content-Type: application/json; charset=utf-8` explicitly (bypassing `axum::Json`'s default header to do so — see the handler's inline comment), verified via `curl` against the live server; (2) `export.ps1` no longer uses `Invoke-RestMethod` at all — it uses `Invoke-WebRequest -UseBasicParsing`, reads `RawContentStream` as raw bytes, and decodes with `[System.Text.Encoding]::UTF8` itself, so it's correct independent of any server's headers.
  - **Finding (2026-07-12, continued further)**: separately, `export.ps1`'s own source file had `上班`/`下班` written as literal characters — Windows PowerShell 5.1 reads a BOM-less UTF-8 `.ps1` file using the system codepage, which corrupts non-ASCII literals *in the script source itself* (a completely different mechanism from, and in the opposite BOM direction to, the output-file requirement). Fixed by building those two words from Unicode code points (`[char]0x4E0A + [char]0x73ED` etc.) instead of literals, making the script's correctness independent of how the `.ps1` file itself is saved/transferred. Documented all three PowerShell 5.1 gotchas (TLS, HTTP response charset, script-source encoding) in README.md so they don't get silently reintroduced by a future edit.
